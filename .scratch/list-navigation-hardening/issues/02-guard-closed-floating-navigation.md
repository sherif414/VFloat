## What to build

Prevent mounted-but-closed floating surfaces from mutating collection navigation state when they receive keydown events. A closed floating surface should not respond to Home, End, or arrow navigation, while open surfaces should keep the current keyboard behavior and Tab should still close an open surface without preventing normal page focus movement.

## Acceptance criteria

- [x] Keydown events on a mounted floating element do not change `activeValue` while the floating surface is closed.
- [x] Home, End, and arrow keys still update navigation state while the floating surface is open.
- [x] Tab still closes an open floating surface when `closeOnTab` is enabled.
- [x] Tab does not prevent default page focus movement.
- [x] Behavior is covered by focused tests that assert open state, `activeValue`, and default-prevention outcomes.

**Category:** bug
**Summary:** Guard closed floating surfaces from keyboard navigation events to prevent stale state mutations

**Current behavior:**
In `useListNavigation`, the `onFloatingKeyDown` event listener is registered on the floating element (`refs.floatingEl`). While it checks `open.value` for `Tab` handling, it does not guard other keys (like arrow keys, Home, or End) against being processed when `open.value === false`. If a floating element remains mounted in the DOM while closed/hidden, any keydowns received on it could trigger mutations in the collection's active state.

**Desired behavior:**
`onFloatingKeyDown` must ignore all keydown events and immediately return if the floating surface is not open (`open.value === false`). While the surface is open, arrow keys, Home, and End should process as usual. Pressing Tab when `closeOnTab` is true should close the open surface without preventing the default browser focus movement.

**Key interfaces:**

- `onFloatingKeyDown` inside `use-list-navigation.ts` — needs a check to ensure `open.value` is true before processing any navigation keys.
- `FloatingContext` state: `open` ref and `setOpen` method.
